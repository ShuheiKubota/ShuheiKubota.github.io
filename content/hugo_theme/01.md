## 背景

Hugo ではコンテンツとテーマが明確に分かれており、テーマはサイトの見た目を定義するうえで重要な仕組みです。

できることが多い反面、ファイルの構造や役割、記述の仕方を理解していないと、どこから手を付けてよいか迷うことがあります。

---

[Hugo Themes](https://themes.gohugo.io/) にはたくさんのテーマが公開されています。
ここから自身のサイトに合うものを選択する、もしくはその内容をもとに (コードを) カスタマイズしていく方法もあります。

ただ、公開されているテーマは汎用的に作られていることもあり、「テーマを作成を作成していきたい」という観点からすると固有の設定があったり余分な機能があったりと、より難易度が高くなることもあります。

そのため、この連載では、いちから作成することで前提知識を少なくしつつ理解を深められるようにしていきたいと思っています。

テーマ作成についての理解をすれば、既存のテーマをカスタマイズするうえでどこに手を入れれば良いか、ということが理解しやすくなります。


## 方針

いちからテーマを作成する流れを通じて公式ドキュメントの紹介やその内容の簡単な解説をし、理解の助けになることを目的としています。

このサイト向けに作っている (現在進行形) テーマを作る過程で学んだ内容を中心に説明していくことになります。

> [!warning]
> テーマ作成のすべての要素に触れるわけではありません。  
> また、実装した機能のうち、その一部を題材として説明をしていきます。
> 
> (かつてこのサイトで公開していた記事を、最近の Hugo の仕様に合わせて編集しなおしたものです)

## その他、知っているとよりよく理解できそうな事柄

### テーマとは

Hugo のドキュメントを当たってみましょう。

> theme
>
> A theme is a module that delivers a complete set of **components defining a site’s layout, presentation, and behavior**. While every theme is a module, not every module is a theme.

引用元: [Glossary](https://gohugo.io/quick-reference/glossary/#theme) (太字は引用者によるものです)

テーマ = 太字の部分、というわけではないのですが、まあ、コンポーネントとは何ぞやというのが分かれば「どういったものを作ればテーマが出来上がるのか」が分かるかもしれません。

### コンポーネントとは

これも Hugo のドキュメントを当たってみます。
最初の英文はさておき、表を見てください。

> component
>
> A component is a collection of related **files**, housed within the unified file system, that fulfills a specific function in building a Hugo project. These components are categorized into seven types: archetypes, assets, content, data, templates, translation tables, and static files, and can be defined within the project or provided by modules. Each component has a dedicated directory within the unified file system:
| Component | 用途 | Directory within the unified file system |
|-|-|-|
| archetypes | コンテンツのひな形 | archetypes |
| assets | js, css 等。[asset pipeline](https://gohugo.io/hugo-pipes/introduction/)での処理向け | **assets** |
| content | コンテンツ | content |
| data | CSV等、レイアウトを駆使して表示するデータ | data |
| templates | レイアウトの定義 | **layouts** |
| translation tables | 語句の翻訳(対応表) | i18n |
| static files | 変換後の `public` にコピーされるファイル | static |

引用元: [Glossary](https://gohugo.io/quick-reference/glossary/#component) (太字は引用者によるものです。用途列も引用者により追加されています)

とくにいじるのは、layouts になります。

もちろん、テーマによっては他の部分も必要となりますが、ここでは特に見た目の部分を重視していますので。

### unified file system とは

上の引用で出てきた "unified file system" についてです。
とはいっても、私はざっくりとしか理解していませんが…

`hugo` コマンドでサイトを生成する際、テーマが提供する上記のファイルとプロジェクトで作成したファイルが ディレクトリーごとに混ぜられ最終的な成果物のもととして使われる、ということを言っています。

なのでたとえばテーマが content を定義していた場合、プロジェクトの content に混ざって公開されることになります。

また、複数のテーマを使った場合 (そういうことが可能なんです) は、互いの内容が優先順位に基づいて適用されることになります。
